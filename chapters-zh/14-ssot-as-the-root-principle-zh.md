## 第 14 章：SSOT 是软件工程的根原则

> 如果把所有软件工程原则继续向下归并，它们最终都在谈 Single Source of Truth。

> 完整公开版（含图）：[SSOT：为什么所有软件工程原则最终都在谈同一件事](https://liminge.space/cn/blog/ssot-nine-layer-pyramid)。

AI-Native 团队最难的问题，不是生成更多代码、更多文档、更多测试或更多计划，而是判断哪个事实才是当前事实。

一个成熟项目里会同时存在大量资产：

- 代码
- 测试
- 日志
- PRD
- 技术设计文档
- 发布说明
- 会议纪要
- 运行手册
- prompt
- agent 记忆
- 对外宣传稿

这些资产写下来的时候可能都是真的。但产品战略会变，业务重心会变，实现细节会变。没有 Single Source of Truth，系统会慢慢变成一组互相矛盾的历史快照。

所以 SSOT 不是简单的文档习惯，而是软件工程原则背后的根原则。

### 1. 为什么其他原则都可以归并到 SSOT

许多熟悉的软件工程原则，都是 SSOT 的不同侧面：

- **DRY** 防止同一个事实在多处代码里重复出现。
- **KISS** 让事实路径足够短，短到人和 AI 都能跟上。
- **SOLID** 稳定责任归属，避免一个模块同时持有无关事实。
- **领域建模** 给业务概念一个权威表达。
- **测试** 给预期行为提供可执行事实。
- **可观测性** 给线上状态提供可验证事实来源。
- **CICD** 给版本能否合并、发布、可信提供统一运行事实。
- **文档治理** 决定哪类事实由哪份文档拥有。

表面不同，底层问题相同：

> 当两个资产互相冲突时，团队应该相信谁？

如果团队回答不了这个问题，所有工程原则最后都会退化成风格偏好。

### 2. 人类记忆无法承载复杂系统

小团队早期经常依赖共享记忆。大家都知道哪个功能是核心，哪个脚本是临时的，哪个测试已经过时，哪份文档是旧的。

这只在系统很小时成立。

随着产品增长，团队会反复重新发现旧工作。两个工程师在不同地方解决同一个问题。某个历史阶段的特殊实现，在产品迁移后仍留在仓库里。测试体系还在保护 legacy 工作流，而真实发布风险已经迁移到别处。

失败模式不是“没有文档”，而是更糟糕的状态：到处都有文档，但没有一个权威的当前事实。

AI-Native 会放大这个压力。AI agent 会产生大量架构说明、代码、测试、日志、会议总结、营销文案和任务报告。这些资产只有在拥有归属和 evidence 时才有价值，否则会成为新的混乱来源。

对 AI agent 来说，SSOT 是行动边界。如果旧战略、新代码、过期日志、未关闭任务和对外文案看起来都同样权威，agent 就会综合出一个看似合理、实际错误的系统。它可能继续保护 legacy，把 POC 过度包装成承诺，或者把多个历史阶段的“当前优先级”合成一个错误计划。

对非开发同事来说，SSOT 是协作入口。产品需要当前战略，运营需要真实 workflow，销售需要承诺边界，客服/支持需要能进入用户问题的日志、evidence 和修复路径。

所以 SSOT 不是研发内部的文档卫生，而是 AI agent、产品、运营、销售、客服/支持、发布 operator 和研发共享的事实系统。

### 3. “当前优先级”是最危险的词

制造脑分裂最简单的方式，就是让多份文档同时描述当前优先级。

README 写一版，路线图写一版，方案文档写一版，发布说明写一版，会议纪要再写一版。

它们可能在不同时间都是真的。

人类读起来需要猜哪一份最新。AI 更麻烦，它可能把所有看起来权威的文本都当成上下文。

规则应该非常严格：

- 当前优先级只能有一个 SSOT。
- 当前负责人只能有一个 SSOT。
- 当前资源竞争只能有一个 SSOT。
- 其他文档可以链接这个来源，但不应该复制一份。

每一句被复制的“当前”，都会变成未来的 bug。

### 4. 产品决策不会自动传播

团队可以决定产品核心发生变化，但这个决策本身不会自动更新代码库。

产品重心切换后，仓库和公司资产里仍然会留下旧语言：

- 旧代码命名
- 旧文档承诺
- 旧发布检查
- 旧测试覆盖
- 旧日志和错误文案
- 旧对外定位
- 旧 agent 指令

这就是为什么产运研必须共享一套事实模型。一个战略决策必须沿着承诺、Flow、领域模型、边界、contract、实现和 evidence 一路穿透。

明确从 A 切到 B 还不是最难的情况。更难的是渐进变化：

- 小 POC 变成内部能力
- 内部能力变成一等领域概念
- 领域概念变成战略核心
- 战略核心变成使命组成部分
- 曾经的战略核心慢慢变成 legacy

没有 SSOT，团队资产会停留在这个过程的不同阶段。

### 5. 测试体系是一个 SSOT 案例

本宣言早期版本强调，测试案例库是唯一可靠、可复利的工程资产。

这个判断仍然成立，但需要补充一个前提：测试体系只有在证明当前产品事实时才可靠。

legacy 测试可以一直是绿的，但新主线完全没有被证明。一次发布可以通过 CICD，却在真实用户路径、AI skills 调用、runtime contract、计费路径、provider 权限或 release evidence 上断裂。

AI agent 可能把这种绿色状态误读为当前主线已经被覆盖。运营可能以为新 workflow 可以跑，销售可能以为能力可以承诺，客服/支持可能继续沿着旧日志和旧排查路径处理用户问题。

成熟的问题不是“我们有没有足够多测试”。

成熟的问题是：

> 这个检查证明了哪个当前产品事实？

测试、日志、状态命令、release evidence、线上检查都是 evidence。产品重心迁移时，evidence 也必须迁移。

### 6. AI-Native 版本的 SSOT

AI 是无状态的。每一次 session 都通过 context 重新进入系统。

这意味着 SSOT 不只是服务人类，它是 AI-Native 工程的记忆架构。

AI agent 需要知道：

- 当前产品身份是什么
- 当前战略是什么
- 哪些能力已经公开承诺
- 哪些 workflow 是真实的
- 哪些领域概念是稳定的
- 哪些 contract 是可执行的
- 哪些实现存在
- 哪些 evidence 能证明它们

如果仓库里存在大量互相冲突的答案，AI 就会综合出一个看似合理、实际错误的系统。

产品、运营、销售、客服/支持也需要同一套模型来判断当前战略、可依赖承诺、真实 Flow 和 evidence。否则非开发工作就会退回到口头记忆和最近一次会议。

SSOT 是团队让无状态智能读懂系统、也让整个团队可操作地使用系统的方式。

### 7. 小结

所有软件工程原则最终都指向同一条纪律：

> 每个重要事实都必须有一个权威位置、一个 owner，以及一条 evidence 路径。
